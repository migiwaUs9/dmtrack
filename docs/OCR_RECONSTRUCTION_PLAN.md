# OCR主体カード認識 + 全カードJSONファジーマッチング 設計

## 背景

現在のシステムは **pHash（画像ハッシュ）優先 → OCRフォールバック** で動作しているが、デュエプレには**絵違い（プロモ/コラボ/シークレット等）**が大量に存在するため、pHashベースでは「同じカードなのにハッシュが異なる」問題が根本的に解決不可能。

方針転換として、**OCRを主軸**にし、**デュエプレ全カードのマスターJSONを作成**したうえで**ファジーマッチング**でカード名を特定する方向で再設計する。

---

## 現状の問題点

### 問題1: pHashの根本的限界

| 問題 | 詳細 |
|------|------|
| 絵違い対応不能 | 同名カードでもイラストが異なればハッシュが全く別物。デュエプレは1枚のカードに3-5種以上の絵違いがあるものも多い |
| DB肥大化 | 絵違い全パターンを登録する必要があり、管理コストが爆発する |
| 初見カード問題 | 新カード・未登録絵違いが来ると必ず失敗→OCRフォールバックに依存 |

**結論**: pHashは「知っている絵の高速ショートカット」としては有用だが、主軸にはなり得ない

### 問題2: OCRの精度不足

`card_hashes.json` の現状を見ると、OCRフォールバックの結果が大量にゴミ文字になっている：

```
"時ァィー一一一一ペーペ一マーー一ニー一一一ーー。ーー、、加語間ーーニーーー"
"ングが位導ーー"
"ーー電肘朋チーー"
```

**正しく読めているケースは全体の1割以下**。原因：

| 原因 | 詳細 |
|------|------|
| ゲームフォント | デュエプレ固有のデザインフォント。Tesseractの日本語モデル（jpn.traineddata）は一般書体向け |
| 前処理の不適合 | 単純な二値化（閾値160固定）ではカード名バナーの背景グラデーション・装飾を除去しきれない |
| ROI座標のズレ | `CARD_NAME_REL = (0.42, 0.20, 0.57, 0.24)` — カード拡大表示のアニメーション位置が微妙にズレる |
| PSMの選択 | `--psm 7`（単一行テキスト）は適切だが、テキストに記号（・、！、卍等）が多い日本語のカード名では誤認識しやすい |

### 問題3: ファジーマッチの照合先が不十分

現在 `_all_card_names` は **`decks.json` に登録されたカード（約40枚）** のみ。デュエプレには**2000枚以上**のカードが存在するため：

- 登録外のカードは絶対に補正できない
- 少数のマッチ候補しかないため、誤った候補にマッチしてしまうリスクも高い

### 問題4: ゴミOCR結果がpHash DBに自動登録されている

```python
# OCRで名前が取れたらハッシュDBに自動登録
if card_name and card_name != "(不明)":
    self._hash_db[hash_str] = card_name
```

→ ゴミ文字でも「(不明)」以外なら登録される → **DB汚染が加速**

## 解決策

### フェーズ1: デュエプレ全カードマスターJSON作成

デュエプレのカード情報を網羅したマスターデータを作成する。

#### データソース候補

| ソース | メリット | デメリット |
|--------|----------|------------|
| **デュエプレ公式サイト / カードリスト** | 正規データ、信頼性高い | API非公開の場合スクレイピングが必要 |
| **Game8 / GameWith 等の攻略サイト** | 網羅的、更新が早い | 構造が変わる可能性 |
| **デュエル・マスターズ Wiki** | 最も網羅的 | デュエプレ専用でないカードも含まれる |
| **手動整理** | 確実 | 非現実的（2000枚以上） |

#### JSON構造案

```json
{
  "cards": [
    {
      "id": "BD01-001",
      "name": "ボルシャック・ドラゴン",
      "name_normalized": "ボルシャックドラゴン",
      "cost": 6,
      "civilization": ["火"],
      "card_type": "クリーチャー",
      "race": ["アーマード・ドラゴン"],
      "power": 6000,
      "rarity": "VR",
      "set": "BASIC第1弾"
    }
  ]
}
```

> [!IMPORTANT]
> **最小限で動作に必要なフィールドは `name` と `name_normalized` のみ**。その他は将来の拡張（コスト/種族フィルタ等）用。まずは名前リストの作成を優先すべき。

#### 正規化ルール

```
原文:       "ヘブンズ・ゲート"
normalized: "ヘブンスケート"  ← 全角→半角、記号除去、長音・濁点正規化
```

- `・` `！` `？` `＝` `卍` → 除去
- `ー` (長音) → 保持（ただしOCR側では `一`(漢字の一) と混同するので両方試す）
- カタカナ・漢字・ひらがなのみ残す

### フェーズ2: OCRパイプラインの改善

#### 2-1. 前処理の強化

```
現在:  resize(5x) → grayscale → threshold(160) → border → invert
改善案: resize(5x) → grayscale → adaptiveThreshold → morphology → border → invert
```

具体的な改善：

| 改善 | 内容 |
|------|------|
| **適応的二値化** | `cv2.adaptiveThreshold()` でローカルな背景グラデーションに対応 |
| **モルフォロジー処理** | `cv2.morphologyEx(MORPH_CLOSE)` でノイズ除去・文字の断裂修復 |
| **色チャンネル分離** | カード名の文字色（白/金/虹）に応じてHSVで分離し、文字部分だけ抽出 |
| **複数閾値の試行** | 閾値を複数パターン試し、OCR結果の信頼度が最も高いものを採用 |

#### 2-2. OCRエンジンの検討

| エンジン | 精度 | 速度 | 備考 |
|----------|------|------|------|
| Tesseract (現行) | △ | ○ | ゲームフォントに弱い |
| **EasyOCR** | ○ | △ | PyTorchベース。日本語の認識精度が高い。GPUで高速化可能 |
| **PaddleOCR** | ◎ | ○ | 最も精度が高いとされる。日本語モデルあり |
| **Windows OCR (WinRT)** | ○ | ◎ | OS内蔵。追加インストール不要。日本語は良好 |
| **manga-ocr** | ◎ (特化) | △ | マンガ・ゲーム系フォントに特化したモデル。カード名に最適の可能性 |

> [!TIP]
> **manga-ocr** は日本語の装飾フォント・デザインフォントに特化して学習されており、デュエプレのカード名に最も適している可能性が高い。まずこれを試すべき。

#### 2-3. 信頼度フィルタリング

```python
# OCR結果の品質スコア
def ocr_quality_score(raw_text: str) -> float:
    jp_chars = len(re.findall(r'[\u3040-\u9fff]', raw_text))
    total = len(raw_text)
    if total == 0:
        return 0.0
    # 日本語文字の割合 × 文字数の適正さ
    jp_ratio = jp_chars / total
    length_score = min(len(raw_text), 20) / 20  # 20文字程度が上限
    return jp_ratio * length_score
```

- スコアが低い場合はDBに登録しない（ゴミ防止）
- 複数フレームの結果を比較し、一貫性のあるもののみ採用

### フェーズ3: ファジーマッチングの強化

#### 3-1. マッチングパイプライン

```
OCR生テキスト
  ↓ ① 正規化（記号除去、全角→半角、長音統一）
  ↓ ② 完全一致チェック（正規化済み全カードDBと照合）
  ↓ ③ 部分一致チェック（OCR結果がカード名を含む / カード名がOCR結果を含む）
  ↓ ④ ファジーマッチ（編集距離 or SequenceMatcher）
  ↓ ⑤ N-gramマッチ（文字レベル2-gram/3-gramの一致率）
  ↓ ⑥ 信頼度チェック → 閾値以下なら "(不明)" として保留
  ↓
カード名確定
```

#### 3-2. 改善された `fuzzy_correct`

```python
def fuzzy_correct_v2(ocr_text: str, all_cards: list[dict],
                     threshold: float = 0.55) -> tuple[str | None, float]:
    """
    全カードマスターDB（2000枚+）に対するファジーマッチ。
    
    Returns: (カード名, 信頼度スコア) or (None, 0.0)
    """
    n_ocr = normalize(ocr_text)
    if len(n_ocr) < 2:
        return None, 0.0
    
    candidates = []
    
    for card in all_cards:
        n_card = card["name_normalized"]
        
        # 完全一致
        if n_ocr == n_card:
            return card["name"], 1.0
        
        # 部分一致（ボーナス付き）
        if n_card in n_ocr or n_ocr in n_card:
            overlap = min(len(n_ocr), len(n_card)) / max(len(n_ocr), len(n_card))
            candidates.append((card["name"], overlap + 0.2))
            continue
        
        # スライドウィンドウ + SequenceMatcher
        score = sliding_window_match(n_ocr, n_card)
        if score >= threshold:
            candidates.append((card["name"], score))
    
    if not candidates:
        return None, 0.0
    
    # 最高スコアの候補を返す
    candidates.sort(key=lambda x: x[1], reverse=True)
    return candidates[0]
```

#### 3-3. コスト/文明による絞り込み（将来拡張）

カード拡大表示時にはコスト（左上の数字）や文明（枠の色）も見えている。これらを補助情報として使えば候補を大幅に絞り込める：

```
OCR候補: ["ボルシャック・ドラゴン", "ボルシャック・ファイター", "ボルメテウス・ホワイト・ドラゴン"]
コスト検知: 6 → "ボルシャック・ドラゴン" (コスト6) に絞り込み
```

## pHashの扱い

> [!WARNING]
> pHashを完全に廃止するのではなく、**「確認済み高信頼キャッシュ」として補助利用**することを推奨。

### 新しいフロー

```
カード拡大表示検知
  ↓
pHash DB照合 ← 高信頼エントリのみ（人間がラベル確認済み）
  ↓ ヒット → カード名確定（高速）
  ↓ ミス ↓
OCR実行
  ↓
全カードDB ファジーマッチ
  ↓
信頼度チェック
  ↓ 高信頼 → カード名確定 + pHash DB に登録
  ↓ 低信頼 → "(不明)" + 画像保存（後で手動ラベル付け）
```

## 現行card_hashes.jsonの評価

現在283エントリ中、正しいカード名が登録されているのは以下の少数のみ：

- `モモキングRX`, `メンデルスゾーン`, `マジック・ソング`, `エメラル`, `フェアリー・ライフ`,
  `ヴォルグ・サンダー`, `ガチンコ・ルーレット`, `ドラゴンズ・サイン`, `水晶の祈り`,
  `入団の光魂`, `メタリカ`, `ドンドン吹雪く`

**280エントリ中まともなのは12エントリ程度**（約4%の正答率）。これがOCRの現状精度を端的に示している。

> [!CAUTION]
> 現在のcard_hashes.jsonは**ゴミエントリが96%を占める**ため、まずクリーンアップが必要。自動登録ロジックには必ず品質フィルタを追加すべき。
